理解模块很重要，但是我发现大多数文档常常会让你在这个问题上摸不着头脑。

来自Python还是Javascript？

大致上，mod.rs有点像__init__.pyPython 或index.jsJavaScript。但只是有点像。这在 Rust 中稍微复杂一些。

Rust 有所不同

文件夹还不能立即用作 Rust 中的模块。

您必须添加一个以mod.rs文件夹命名的文件，才能显示与该文件夹同名的新模块。 文件夹中的代码mod.rs是该模块的内容。 文件夹中的所有其他文件可能依次显示为子模块（更多信息见下文）。

等等，还有其他方法

您可以添加与文件夹同级命名的mod.rs文件，而不是在文件夹中添加文件。<folder_name>.rs

正如 MarkusToman 在评论中指出的那样，这是自 rustc 1.30 以来的首选方式。

来自Rust 参考：

注意：在 rustc 1.30 之前，使用 mod.rs 文件是加载嵌套子模块的方法。建议使用新的命名约定，因为它更一致，并且避免在项目中有多个名为 mod.rs 的文件。

完整示例

src
    utils
        bar.rs
        foo.rs
    main.rs

此时，编译器还不知道src/utils/foo.rs和src/utils/bar.rs。

首先，你必须暴露src/utils/。如上所示，你有两个选择：

  • 添加文件：src/utils/mod.rs
  • 添加文件src/utils.rs（名称与文件夹完全相同，但没有扩展名）

现在，相对于 src 文件夹（又名 crate 级别），有一个名为的模块utils可用。

其次，您必须公开文件src/utils/foo.rs和src/utils/bar.rs。

为此，模块必须声明 2 个以这些文件命名的新子模块。因此（或）utils的内容应为：src/utils/mod.rssrc/utils.rs

pub mod bar;
pub mod foo;

现在，这两个文件中公开的内容都可以在其他模块中使用了！🎉

您也可以写入以下内容src/main.rs：

mod utils;
use utils::{foo, bar};

生成的文件结构

选项 1 •mod.rs（旧方法）：

src
    utils
        bar.rs
        foo.rs
        mod.rs
    main.rs

选项 2 •<folder_name>.rs（首选方式）：

src
    utils
        bar.rs
        foo.rs
    utils.rs
    main.rs


有关模块如何工作的更多高级细节

这仍然是一个表面解释，你的下一个目的地是官方文档🧑‍🎓

还有第三种方法来声明模块（核心语言）：

mod utils {
  // module code goes here. That's right, inside of the file.
}

但也可以只写mod utils;。在这种情况下，如上所示，Rust 知道搜索src/utils.rs或src/utils/mod.rs。

瞧，当您尝试在文件中使用模块时（src/main.rs例如），您可以通过以下方式引用它：

  • 从内部：src/main.rs
    • mod module { ... }
  • 来自嵌套模块内部：src/main.rs
    • mod module { pub mod sub_module { ... } }
  • 来自 sybling 文件：src/*.rs
  • 来自mod.rssybling 文件夹中的文件：src/*/mod.rs
  • （以及上述的无限递归组合）

包含的文件或文件夹mod.rs不会成为模块。
相反，Rust 语言允许您使用文件层次结构来组织模块（语言功能）。

真正有趣的是，您可以自由地将所有方法混合在一起。

例如，你可能认为你不能直接src/utils/foo.rs从引用main.rs。
但你可以：

// src/main.rs
mod utils {
  pub mod foo;
}

重要笔记：

  • 文件中声明的模块始终优先（因为您实际上永远不需要搜索文件层次结构）
  • 你不能使用其他两种方法来引用同一个模块

例如，同时拥有src/utils.rs和src/utils/mod.rs将在编译时引发以下错误：

error[E0761]: file for module `utils` found at both "src/utils.rs" and "src/utils/mod.rs"
 --> src/main.rs:1:1
  |
1 | mod utils;
  | ^^^^^^^^^^
  |
  = help: delete or rename one of them to remove the ambiguity

让我们总结一下。模块暴露给编译器：

  • 从上到下
  • 仅供参考（这就是为什么在你的模块被“导入”之前你没有智能感知的原因）
  • 从入口点开始（默认情况下是src/main.rs或src/lib.rs。但它可能是您在中配置的任何内容Cargo.toml。但这与这个问题关系不大）

根据我们前面的例子，我们得到：

  1. src/main.rs->crate

    由于crate模块包含mod utils;我们接下来得到：

  2. src/utils.rs或src/utils/mod.rs->crate::utils

    由于utils模块包含mod foo;我们接下来得到：

  3. src/utils/foo.rs->crate::utils::foo

想象一下以下目录结构：

代码/
  `- main.rs
   - 某物/
     `- mod.rs

如果main.rs您这样做mod something;，那么它将在something/mod.rs文件中查找要用作模块声明的内容something。

另一种方法是将目录something.rs中的某个文件放进去code/。

回顾一下，当你编写一个空的模块声明（例如）时mod something;，它看起来如下：

  • something.rs同一目录中的文件
  • 在同一目录mod.rs中名为文件夹的文件something

然后，它使用其中一个文件的内容作为模块声明的内容。

rs除了始终与模块匹配的 lib.rs 和 main.rs 之外，每个文件crate都有自己的模块。

声明模块只有一种方法：

/* pub */ mod sub_module1;

模块不能在根/包模块树之外声明（即，沿着模块树向上，子模块必须始终具有直接在lib.rs或中声明的父模块main.rs，因此第一个程序子模块必须始终在那里声明 - 如果还不够明显的话，则为树数据结构）。

有两种方法可以将模块嵌套在声明它的模块内：

  • 在<module_where_it_is_declared>/<module_name.rs>
  • 在<module_where_it_is_declared>/module_name/mod.rs

如果module_where_it_is_declared是 crate 模块，那么这个相应的子文件夹就不需要了（从上面的方案中消失）。

这是一个对 lib 和二进制包都有效的例子：

src
|---lib.rs ( contains: pub mod b2; )
|---b2.rs ( contains: pub mod bb2; )
|---b2
|   |---bb2.rs
.   .

或者：

src
|---lib.rs ( contains: pub mod b2; )
|---b2
|   |---mod.rs ( contains: pub mod bb2; )
|   |---bb2.rs
.   .

您可以看到，您可以混合搭配（b2 使用mod.rs方式，bb2 使用“文件”方式）。

以下是仅使用同样有效的文件模式的一种方法：

src
|---lib.rs ( contains: pub mod b2; )
|---b2.rs ( contains: pub mod bb2; )
|---b2
|   |---bb2.rs (contains: pub mod bbb2; )
|   |---bbb2.rs (contains: pub mod bbbb2; )
|   |---bbb2
|   |   |---bbbb2.rs
.   .   .

我猜这取决于你想如何嵌套模块。

我喜欢mod.rs模块的语法，它只导出其他子模块，而没有任何其他（或很少）代码，尽管你可以在其中放置任何你想要的内容mod.rs。

我使用mod.rs类似于 JS/TS 世界中的桶模式，将多个子模块卷入单个父模块中。

还不要忘记，可以通过添加范围块来内联定义（不仅仅是声明）模块：

pub mod my_submodule {
    // ...
}
