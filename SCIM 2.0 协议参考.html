      Okta 和 SCIM 版本 2.0

      本参考重点介绍 Okta API 端点如何与跨域身份管理系统 (SCIM) 特定 API 调用共享信息。

      本文档专门介绍SCIM 规范的2.0 版。有关 SCIM 规范的 1.1 版，请参阅我们的SCIM 1.1 参考。

      SCIM 协议是一种应用级 REST 协议，用于在 Web 上配置和管理身份数据。该协议支持核心身份资源的创建、发现、检索和修改。

      为了更好地理解 SCIM 以及使用 Okta 的 SCIM 具体实现，请参阅我们的《理解 SCIM》指南或博客文章《什么是 SCIM？》。

      注意： Okta 按照互联网工程任务组的 RFC 文档中规定的方式实现了 SCIM 2.0：

      • 定义、概述、概念和要求：RFC 7642
      • 核心架构：RFC 7643
      • 协议：RFC 7644

      SCIM 用户操作

      创建用户

      用户创建操作将用户的应用程序配置文件从 Okta 移交给服务提供商作为用户对象。用户的应用程序配置文件表示添加用户对象时在配置文件选项卡上定义的键值属性。

      要启用用户配置，您必须在管理控制台中配置配置选项：

      1. 从 Okta 组织的集成列表中选择您的 SCIM 集成。
      2. 单击 “Provisioning”选项卡 下的 “To App” 和 “Edit” 。
      3. 单击 “启用” ，然后 在 “创建用户”部分中单击“保存” 。

      有关启用 SCIM 集成的配置功能的更多信息，请参阅配置应用程序集成的配置。

      完成此步骤后，每当将用户分配到 Okta 中的集成时，都会向 SCIM 服务器发出以下请求：

      • 确定用户对象是否存在。Okta 针对 SCIM 服务器上存储的值运行查询 userName 。如果查询与用户对象匹配，则 SCIM 服务器将返回用户对象的唯一 ID 值。Okta 将此值存储为 externalId Okta 用户配置文件中的值。
      • 如果在 SCIM 服务器上找不到该用户，则创建该用户。
      • 如果在 SCIM 服务器上找到用户，但 Okta 帐户未激活，请在 Okta 中激活该用户。
      • 如果在 SCIM 服务器上找到用户并且 Okta 帐户处于活动状态，则通过设置其唯一 externalId 值来更新 Okta 配置文件，以匹配从 SCIM 服务器返回的 ID 值。

      判断用户是否已经存在

      获取/用户

      Okta 通过带有查询参数的 GET 方法请求检查 SCIM 服务器上是否存在用户对象filter=userName eq "{userName}"。您的 SCIM 服务器必须支持此查询参数才能成功为用户提供 Okta。此检查使用 (等号) 运算符针对为 SCIM 集成配置的唯一标识符执行eq。

      例如，如果您将电子邮件属性配置为唯一标识符，则确定用户是否存在的查询参数是filter=userName eq "{email}"。

      注意：过滤器必须检查服务提供商配置文件中所有用户的唯一属性。

      对于 Okta 集成网络 (OIN) 集成，此过滤器在提交过程中由指定的 Okta 应用分析师协助配置。集成提交通过OIN 管理器处理。

      Okta 向服务提供商发出的请求形式如下：

      GET /scim/v2/Users?filter=userName%20eq%20%22test.user%40okta.local%22&startIndex=1&count=100 HTTP/1.1
User-Agent: Okta SCIM Client 1.0.0
Authorization: <Authorization credentials>

      SCIM 应用程序检查提供的过滤器，如果没有用户符合过滤条件，则返回空响应。例如：

      HTTP/1.1 200 OK
Date: Tue, 10 Sep 2019 01:49:39 GMT
Content-Type: text/json;charset=UTF-8

{
    "schemas": ["urn:ietf:params:scim:api:messages:2.0:ListResponse"],
    "totalResults": 0,
    "startIndex": 1,
    "itemsPerPage": 0,
    "Resources": []
}

      如果没有用户对象符合过滤条件，则 SCIM 应用程序的另一个可接受的响应是返回错误模式响应：

      HTTP/1.1 404 Not Found
Date: Tue, 10 Sep 2019 01:58:03 GMT
Content-Type: text/html; charset=UTF-8

{
    "schemas": ["urn:ietf:params:scim:api:messages:2.0:Error"],
    "detail": "User not found",
    "status": 404
}

      如果 SCIM 服务器确实返回了用户对象，Okta 会自动将结果与 Okta 中的用户进行匹配，并将用户的应用程序配置文件发送到 SCIM 服务器。

      创建用户

      发布/用户

      如果在 SCIM 服务器上找不到 User 对象，则 Okta 会尝试通过包含用户应用程序配置文件的 POST 方法请求来创建它。请求如下所示：

      POST /scim/v2/Users HTTP/1.1
User-Agent: Okta SCIM Client 1.0.0
Authorization: <Authorization credentials>

{
    "schemas": ["urn:ietf:params:scim:schemas:core:2.0:User"],
    "userName": "test.user@okta.local",
    "name": {
        "givenName": "Test",
        "familyName": "User"
    },
    "emails": [{
        "primary": true,
        "value": "test.user@okta.local",
        "type": "work"
    }],
    "displayName": "Test User",
    "locale": "en-US",
    "externalId": "00ujl29u0le5T6Aj10h7",
    "groups": [],
    "password": "1mz050nq",
    "active": true
}

      注意：password即使未启用密码同步，Okta 也会在创建用户请求中发送该参数。此参数充当旧版配置平台的占位符，其值本质上不相关或不敏感。

      SCIM 服务器的响应包含创建的用户对象：

Date: Tue, 10 Sep 2019 02:02:58 GMT
Content-Type: text/json;charset=UTF-8

{
    "schemas": ["urn:ietf:params:scim:schemas:core:2.0:User"],
    "id": "23a35c27-23d3-4c03-b4c5-6443c09e7173",
    "userName": "test.user@okta.local",
    "name": {
        "givenName": "Test",
        "familyName": "User"
    },
    "emails": [{
        "primary": true,
        "value": "test.user@okta.local",
        "type": "work"
    }],
    "displayName": "Test User",
    "locale": "en-US",
    "externalId": "00ujl29u0le5T6Aj10h7",
    "active": true,
    "groups": [],
    "meta": {
        "resourceType": "User"
    }
}

      另一种情况是，如果您的 SCIM 服务器具有您想要为任何新用户添加的自定义属性。在创建用户时，用户配置文件的应用程序架构中定义的任何自定义属性都将应用于用户的应用程序配置文件。对 SCIM 服务器的请求如下所示：

      POST /scim/v2/Users HTTP/1.1
User-Agent: Okta SCIM Client 1.0.0
Authorization: <Authorization credentials>

{
    "schemas": ["urn:ietf:params:scim:schemas:core:2.0:User"],
    "userName": "test.user@okta.local",
    "name": {
        "givenName": "Test",
        "familyName": "User"
    },
    "emails": [{
        "primary": true,
        "value": "test.user@okta.local",
        "type": "work"
    }],
    "displayName": "Test User",
    "locale": "en-US",
    "groups": [],
    "password": "1mz050nq",
    "active": true
}

      SCIM 服务器的响应包含创建的用户对象及其附加自定义属性：

      HTTP/1.1 201 Created
Date: Tue, 10 Sep 2019 02:02:58 GMT
Content-Type: text/json;charset=UTF-8

{
    "schemas": ["urn:ietf:params:scim:schemas:core:2.0:User"],
    "id": "23a35c27-23d3-4c03-b4c5-6443c09e7173",
    "userName": "test.user@okta.local",
    "name": {
        "givenName": "Test",
        "familyName": "User"
    },
    "emails": [{
        "primary": true,
        "value": "test.user@okta.local",
        "type": "work"
    }],
    "displayName": "Test User",
    "locale": "en-US",
    "externalId": "00ujl29u0le5T6Aj10h7",
    "active": true,
    "userType": "Contractor"
    "groups": [],
    "meta": {
        "resourceType": "User"
    }
}

      在此示例中，externalID和userType属性未包含在原始 POST 方法请求中，但在 SCIM 服务器响应中生成并返回。

      注意：如果您的自定义属性是在 Okta 集成中定义的（作为应用程序到 Okta 的映射），则自定义属性不会应用于 Okta 用户配置文件，直到管理员从 SCIM 应用程序运行导入或强制同步操作。

      如果 SCIM 服务器向配置请求返回一个空的响应主体，那么 Okta 会将该操作标记为无效，并且管理控制台会显示错误：

      “将用户自动配置userName到应用程序AppName失败：创建用户时出错displayName：创建新用户返回空用户。”

      如果 Okta 尝试创建的用户对象存在于服务提供商应用中，则服务提供商需要使用错误架构进行响应以停止配置作业。响应如下所示：

      HTTP/1.1 409 Conflict
Date: Tue, 10 Sep 2019 02:22:30 GMT
Content-Type: text/json;charset=UTF-8

{
    "schemas": ["urn:ietf:params:scim:api:messages:2.0:Error"],
    "detail": "User already exists in the database.",
    "status": 409
}

      检索用户

      获取/用户

      从 SCIM 服务器导入用户对象时，Okta 会访问/Users端点并逐页处理它们，使用startIndex、count和totalResults作为分页引用。同样，在返回大量资源列表时，您的 SCIM 实现必须支持分页。使用结果限制count和偏移量会startIndex在请求中返回较小的资源分组。

      注意：itemsPerPage、和startIndex值totalResults需要作为整数交换，而不是字符串。

      Okta 使用count=100分页引用来返回最多 100 个元素。如果 的值totalResults大于 100，那么在 Okta 检索完前 100 个资源后， 将startIndex成为，并作为查询参数与 一起在新请求中startIndex+100传递给端点。此分页操作重复进行，直到查看完所有页面。count/Users

      SCIM 服务器必须始终按相同的顺序返回请求的结果，无论为count和startIndex分页引用提供了哪些值。有关分页的更多信息，请参阅V2.0 规范的第 3.4.2.4 节。

      以下是 Okta 从 SCIM 应用程序检索用户的示例请求：

      GET /scim/v2/Users?startIndex=1&count=100 HTTP/1.1
User-Agent: Okta SCIM Client 1.0.0
Authorization: <Authorization credentials>

      对此请求的响应是 SCIM 应用程序中找到的所有资源的 JSON 列表。

      更新特定用户 (PUT)

      更新用户对象是指修改 Okta 用户应用程序配置文件中映射到 SCIM 应用程序中的属性的属性。

      要更新用户对象，您需要在管理控制台中启用该功能：

      1. 从 Okta 组织的集成列表中选择您的 SCIM 集成。
      2. 在 “Provisioning” 选项卡下，单击 “To App” 。
      3. 在 更新用户属性 选项中，单击 启用 ，然后 单击保存 。

      检索用户

      获取/用户/ {用户 ID}

      要更新用户，Okta 首先发出 GET 方法请求/Users/{userID}并检索 User 对象的主体：

      更新用户

      PUT /用户/ {用户ID}

      笔记：

      • 对于任何新的 OIN 应用程序集成，对用户对象的所有更新都使用 PUT 方法请求进行处理，除非在 更新特定用户 (PATCH)中另有说明 。
      • 对于使用应用程序集成向导 (AIW) 创建的任何自定义应用程序集成，对用户对象的所有更新都使用 PUT 方法请求来处理。

      SCIM 服务器检索到用户对象后，Okta 会修改已更改的属性，并使用新主体向/Users/{userID}端点运行 PUT 方法请求：

      更新特定用户 (PATCH)

      PATCH /用户/ {用户ID}

      对于新的 OIN 应用程序集成，以下操作通过 PATCH 方法请求更新用户对象：

      • 激活用户
      • 停用用户
      • 同步用户密码

      对用户对象的所有其他更新都通过 PUT 方法请求来处理。

      对于使用 AIW 创建的任何自定义应用程序集成，所有更新用户对象的 SCIM 操作（包括这些操作）始终通过 PUT 方法请求发送。

      Okta 用户配置文件中的属性active代表用户的当前状态。

      要停用用户，您需要在管理控制台中启用以下功能：

      1. 从 Okta 组织的集成列表中选择您的 SCIM 集成。
      2. 单击 “Provisioning”选项卡 下的 “To App” 和 “Edit” 。
      3. 单击“停用用户”部分 中的 “启用” ，然后 单击“ 保存” 。

      当用户被停用时，Okta 会发送此请求：

      PATCH /scim/v2/Users/23a35c27-23d3-4c03-b4c5-6443c09e7173 HTTP/1.1
User-Agent: Okta SCIM Client 1.0.0
Authorization: <Authorization credentials>

{
    "schemas": ["urn:ietf:params:scim:api:messages:2.0:PatchOp"],
    "Operations": [{
        "op": "replace",
        "value": {
            "active": false
        }
    }]
}

      SCIM 服务器的响应需要是更新的用户对象：

      HTTP/1.1 200 OK
Date: Tue, 10 Sep 2019 03:50:23 GMT
Content-Type: text/json;charset=UTF-8

{
    "schemas": ["urn:ietf:params:scim:schemas:core:2.0:User"],
    "id": "23a35c27-23d3-4c03-b4c5-6443c09e7173",
    "userName": "test.user@okta.local",
    "name": {
        "givenName": "Another",
        "middleName": "",
        "familyName": "User"
    },
    "emails": [{
        "primary": true,
        "value": "test.user@okta.local",
        "type": "work",
        "display": "test.user@okta.local"
    }],
    "active": false,
    "groups": [],
    "meta": {
        "resourceType": "User"
    }
}

      注意： SCIM 服务器对 PATCH 方法请求的响应也可能是 HTTP 204 响应，不返回任何正文。

      删除用户

      删除/用户/ {用户ID}

      Okta 不会对 SCIM 应用程序中的用户对象执行 DELETE 操作。

      如果用户在 Okta 内部的集成中被停用或删除，Okta 会向您的 SCIM 应用发送请求以将属性设置active为false。对于在 Okta 内部被暂停的用户，不会发送取消配置事件。

      • 对于所有新的 OIN 应用程序集成，更新用户对象的请求是通过 PATCH 方法请求发送的。
      • 对于使用 AIW 创建的任何自定义应用程序集成，此请求都是通过 PUT 方法请求发送的。

      有关删除用户的详细说明，请参阅删除（取消配置）。

      SCIM 集团运营

      创建组

      POST /群组

      要在 SCIM 服务器上创建组对象，首先需要在管理控制台中启用组推送功能的配置：

      1. 从 Okta 组织的集成列表中选择您的 SCIM 集成。
      2. 单击 “推送组”选项卡 上的 “推送组” 。

      您可以通过指定名称或规则来选择要推送的现有 Okta 组。如果不存在组，请在 Okta 中创建一个组，然后将其推送到 SCIM 服务器。有关更多信息，请参阅Okta 帮助文档中的组推送。

      选择组后，Okta 向服务提供商发出 POST 方法请求：

      POST /scim/v2/Groups HTTP/1.1
User-Agent: Okta SCIM Client 1.0.0
Authorization: <Authorization credentials>

{
    "schemas": ["urn:ietf:params:scim:schemas:core:2.0:Group"],
    "displayName": "Test SCIMv2",
    "members": []
}

      当收到此请求时，SCIM 服务器将使用 Group 对象进行响应，就像对 GET 方法请求一样/Groups/${groupID}/：

      HTTP/1.1 201 Created
Date: Tue, 10 Sep 2019 04:54:18 GMT
Content-Type: text/json;charset=UTF-8

{
    "schemas": ["urn:ietf:params:scim:schemas:core:2.0:Group"],
    "id": "abf4dd94-a4c0-4f67-89c9-76b03340cb9b",
    "displayName": "Test SCIMv2",
    "members": [],
    "meta": {
        "resourceType": "Group"
    }
}

      检索组

      获取/群组

      从 SCIM 服务器导入 Group 对象时，Okta 会访问/Groups端点并逐页处理它们，使用startIndex、count和totalResults值作为参考。同样，在返回大量资源列表时，您的 SCIM 实现必须支持分页。使用结果限制count和偏移量会startIndex在请求中返回较小的资源分组。

      注意：itemsPerPage、和startIndex值totalResults需要作为整数交换，而不是字符串。

      Okta 使用count=100分页引用来返回最多 100 个元素。如果 的值totalResults大于 100，那么在 Okta 检索完前 100 个资源后， 将startIndex成为，并作为查询参数与 一起在新请求中startIndex+100传递给端点。此分页操作重复进行，直到查看完所有页面。count/Groups

      SCIM 服务器必须始终按照相同的顺序返回请求的结果，无论为count和startIndex分页引用提供了哪些值。

      以下是 Okta 从 SCIM 应用程序检索 Group 对象的示例请求：

      GET /scim/v2/Groups?startIndex=1&count=100 HTTP/1.1
User-Agent: Okta SCIM Client 1.0.0
Authorization: <Authorization credentials>

      对此请求的响应是 SCIM 应用程序中找到的所有 Group 对象的 JSON 列表。

      eq您还必须在 SCIM 服务器上使用 (equals) 运算符实现结果过滤。Okta 通过带有filter=displayName eq "{groupName}"path 参数的 GET 方法请求检查 SCIM 服务器上是否存在 Group 对象，其中groupName是目标应用程序上的组名称。

      GET /Groups当您想要更新一个组但不知道外部组 ID 时，Okta 会使用带有组名过滤器的请求。

      注意：当管理员配置配置应用集成并清除导入组复选框时，外部组 ID 未知。请参阅为应用集成配置配置。

      以下是向 SCIM 服务器发出请求的示例：

      GET /scim/v2/Groups?filter=displayName%20eq%20%22Test%20SCIMv2%22&startIndex=1&count=100 HTTP/1.1
User-Agent: Okta SCIM Client 1.0.0
Authorization: <Authorization credentials>

      SCIM 服务器处理该请求并使用下列之一进行响应：

      • 如果群组符合过滤条件，则显示其列表，例如： 
        HTTP/1.1 200 OK
Date: Wed, 15 May 2024 10:02:45 GMT
Content-Type: text/json;charset=UTF-8

{
    "schemas": [
        "urn:ietf:params:scim:api:messages:2.0:ListResponse"
    ],
    "totalResults": 1,
    "startIndex": 1,
    "itemsPerPage": 1,
    "Resources": [
        {
        "id": "e7d09e9b3faa4888b65cf9e9316cba1c",
        "meta": {
            "created": "2024-05-15T09:21:23",
            "lastModified": "2024-05-15T09:21:23",
            "version": "v1.0"
        },
        "displayName": "Test SCIMv1"
       },
    ]
}
      • 如果没有组符合过滤条件，则返回空响应，例如： 
        HTTP/1.1 200 OK
Date: Wed, 15 May 2024 11:02:14 GMT
Content-Type: text/json;charset=UTF-8

{
    "schemas": ["urn:ietf:params:scim:api:messages:2.0:ListResponse"],
    "totalResults": 0,
    "startIndex": 1,
    "itemsPerPage": 0,
    "Resources": []
}

      检索特定组

      获取/群组/ {群组 ID}

      在某些情况下，Okta 需要对特定对象运行 GET 方法请求{groupID}，例如，查看 SCIM 服务器上是否仍存在 Group 对象。请求如下所示：

      GET /scim/v2/Groups/abf4dd94-a4c0-4f67-89c9-76b03340cb9b HTTP/1.1
User-Agent: Okta SCIM Client 1.0.0
Authorization: <Authorization credentials>

      来自服务器的响应是 Group 对象的详细信息：

      HTTP/1.1 200 OK
Date: Tue, 10 Sep 2019 05:06:25 GMT
Content-Type: text/json;charset=UTF-8

{
    "schemas": ["urn:ietf:params:scim:schemas:core:2.0:Group"],
    "id": "abf4dd94-a4c0-4f67-89c9-76b03340cb9b",
    "displayName": "Test SCIMv2",
    "members":  [{
        "value": "b1c794f24f4c49f4b5d503a4cb2686ea",
        "display": "SCIM 2 Group A"
    }],
    "meta": {
        "resourceType": "Group"
    }
}

      更新特定组名称

      PATCH /群组/ {groupID}

      PUT /群组/ {群组ID}

      Okta 群组现有名称的更新由向您的 SCIM 应用发出的方法请求处理。群组对象必须已推送到 SCIM 服务器。

      • 对于所有新的 OIN 应用程序集成，更新 Group 对象的请求是通过 PATCH 方法请求发送的。
      • 对于使用 AIW 创建的自定义应用程序集成，更新 Group 对象的请求是通过 PUT 请求发送的。 
      PATCH /scim/v2/Groups/abf4dd94-a4c0-4f67-89c9-76b03340cb9b HTTP/1.1
User-Agent: Okta SCIM Client 1.0.0
Authorization: <Authorization credentials>

{
    "schemas": ["urn:ietf:params:scim:api:messages:2.0:PatchOp"],
    "Operations": [{
        "op": "replace",
        "value": {
            "id": "abf4dd94-a4c0-4f67-89c9-76b03340cb9b",
            "displayName": "Test SCIMv2"
        }
    }]
}

      每次有群组成员身份更新操作时都会触发群组名称更新操作。

      SCIM 服务器响应返回更新的 Group 对象：

      HTTP/1.1 200 OK
Date: Tue, 10 Sep 2019 05:08:48 GMT
Content-Type: text/json;charset=UTF-8

{
    "schemas": ["urn:ietf:params:scim:schemas:core:2.0:Group"],
    "id": "abf4dd94-a4c0-4f67-89c9-76b03340cb9b",
    "displayName": "Test SCIMv20",
    "members": null,
    "meta": {
        "resourceType": "Group"
    }
}

      注意： SCIM 服务器对 PATCH 方法请求的响应也可能是 HTTP 204 响应，不返回任何正文。

      更新特定群组成员资格

      PATCH /群组/ {groupID}

      PUT /群组/ {群组ID}

      要在 SCIM 服务器上特定推送的 Group 对象内添加或删除用户，Okta 需要以下内容：

      • 用户必须是 Okta 中群组的成员。
      • 该用户已被添加到 管理控制台内 SCIM 集成的 “分配”选项卡下。
      • 该组被推送到 管理控制台内 SCIM 集成的“ 推送组”选项卡下。

      如果满足这三个要求，Okta 会发送请求将指定的用户添加到 SCIM 服务器上的组对象。

      • 对于所有新的 OIN 应用程序集成，PATCH 方法请求用于更新 Group 对象，例如： 
        PATCH /scim/v2/Groups/abf4dd94-a4c0-4f67-89c9-76b03340cb9b HTTP/1.1
User-Agent: Okta SCIM Client 1.0.0
Authorization: <Authorization credentials>

{
    "schemas": ["urn:ietf:params:scim:api:messages:2.0:PatchOp"],
    "Operations": [{
        "op": "remove",
        "path": "members[value eq \"89bb1940-b905-4575-9e7f-6f887cfb368e\"]"
        },
        {
        "op": "add",
        "path": "members",
        "value": [{
            "value": "23a35c27-23d3-4c03-b4c5-6443c09e7173",
            "display": "test.user@okta.local"
        }]
    }]
}
      • 对于使用 AIW 创建的自定义应用程序集成，PUT 方法请求用于更新 Group 对象，例如： 
        PUT /scim/v2/Groups/abf4dd94-a4c0-4f67-89c9-76b03340cb9b HTTP/1.1
User-Agent: Okta SCIM Client 1.0.0
Authorization: <Authorization credentials>

{
    "schemas": ["urn:ietf:params:scim:schemas:core:2.0:Group"],
    "displayName": "Test SCIMv2",
    "members": [
            {
            "value": "23a35c27-23d3-4c03-b4c5-6443c09e7173",
            "display": "test.user@okta.local"
            }
    ]
}

      SCIM 服务器响应返回更新的 Group 对象：

      HTTP/1.1 200 OK
Date: Tue, 10 Sep 2019 05:06:25 GMT
Content-Type: text/json;charset=UTF-8

{
    "schemas": ["urn:ietf:params:scim:schemas:core:2.0:Group"],
    "id": "abf4dd94-a4c0-4f67-89c9-76b03340cb9b",
    "displayName": "Test SCIMv20",
    "members": [
        {
            "value": "23a35c27-23d3-4c03-b4c5-6443c09e7173",
            "display": "test.user@okta.local"
        }
     ],
    "meta": {
        "resourceType": "Group"
    }
}

      在此示例中，该members属性返回一个空值。Okta 不需要返回用户列表，但它需要有关组的其他详细信息。

      您还可以使用该操作将成员资格的完整推送发送到 SCIM 服务器replace。此操作会用提供的对象值替换所有组成员。

      PATCH /scim/v2/Groups/abf4dd94-a4c0-4f67-89c9-76b03340cb9b HTTP/1.1
User-Agent: Okta SCIM Client 1.0.0
Authorization: <Authorization credentials>

{
  "schemas": [
    "urn:ietf:params:scim:api:messages:2.0:PatchOp"
  ],
  "Operations": [
    {
      "op": "replace",
      "path": "members",
      "value": [
        {
          "value": "23a35c27-23d3-4c03-b4c5-6443c09e7173",
          "display": "test.user@okta.local"
        },
        {
          "value": "89bb1940-b905-4575-9e7f-6f887cfb368e",
          "display": "test.user@okta.local"
        }
      ]
    }
  ]
}

      注意： SCIM 服务器对 PATCH 方法请求的响应也可能是 HTTP 204 响应，不返回任何正文。

      删除特定组

      删除/群组/ {groupID}

      Okta 管理员可以从管理控制台的 SCIM 集成下的推送组选项卡下删除推送的组。

      在推送组选项卡上，单击活动，然后单击取消链接推送组。在出现的对话框中，您可以选择是否要在目标应用程序中删除组或将组保留在SCIM 服务器上的目标应用程序中。

      当您选择删除目标应用程序中的组选项并单击取消链接时，Okta 会发送 DELETE 方法请求：

      DELETE /scim/v2/Groups/abf4dd94-a4c0-4f67-89c9-76b03340cb9b HTTP/1.1
User-Agent: Okta SCIM Client 1.0.0
Authorization: <Authorization credentials>

      SCIM 服务器可以返回空响应：

      HTTP/1.1 204 No Content
Date: Tue, 10 Sep 2019 05:29:25 GMT
