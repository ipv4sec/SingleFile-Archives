1.介绍

1.1背景

本节是非规范性的。

IndieAuth 规范最初是作为一种获取 OAuth 2.0 访问令牌供 Micropub 客户端使用的方式。它既可用于获取访问令牌，也可用于对登录任何应用程序的用户进行身份验证。它建立在 OAuth 2.0 框架之上，虽然本文档应该为实施者提供足够的指导，但参考核心 OAuth 2.0 规范可以帮助回答任何剩余的问题。更多信息可以在 IndieWeb wiki 上找到。

1.2OAuth 2.0 扩展

IndieAuth 以 OAuth 2.0 [ RFC6749 ] 框架为基础，具体如下：

  • 指定用户标识符的格式（可解析的 URL）
  • 指定根据用户的个人资料 URL 发现 IndieAuth 元数据（包含授权和令牌端点）的方法
  • 指定客户端 ID 的格式（包含客户端元数据的可解析 URL）
  • 由于客户端 ID 是可解析的 URL，因此无需在授权端点进行客户端注册
  • 重定向 URL 注册通过验证在客户端 ID URL 处获取的数据进行
  • 指定返回授权请求的用户的用户标识符和个人资料信息的机制

此外，OAuth 2.0 定义的参数（特别是state、code和scope）遵循与 OAuth 2.0 附录 A [ RFC6749 ] 定义的相同语法要求。

本文档中的关键词“必须”，“不得”，“要求” ， “应该”，“不应”，“应该”，“不应该”，“推荐”，“可以”和“可选”应按照 [ RFC2119 ]中的描述进行解释。

2.1一致性等级

IndieAuth 实现可以实现 IndieAuth 服务器或客户端的一个或多个角色。本节介绍每个角色的一致性标准。

下面列出了已知的 IndieAuth 实现类型。

2.1.1授权端点

IndieAuth 授权端点负责从最终用户获取身份验证或授权同意并生成和验证授权码。

2.1.2令牌端点

IndieAuth 令牌端点负责生成和验证 OAuth 2.0 Bearer 令牌。

2.1.3Micropub 客户端

Micropub 客户端将尝试根据允许发现授权端点的 URL 获取 OAuth 2.0 Bearer Token，并在发出 Micropub 请求时使用该令牌。

2.1.4IndieAuth 客户端

IndieAuth 客户端是尝试通过允许发现授权端点的 URL 对用户进行身份验证的客户端，但不需要 OAuth 2.0 Bearer Token。

3.标识符

3.1发行者识别符

授权服务器的发行者标识符，主要用于防止授权服务器混淆攻击。发行者标识符是一个使用“https”方案的 URL，没有 [ RFC9207 ] 中定义的查询或片段组件，但是，标识符还必须是 URL 的前缀indieauth-metadata。例如，对于indieauth-metadata端点https://example.com/.well-known/oauth-authorization-server，发行者 URL 可以是https://example.com/，或者对于 的元数据 URL https://example.com/wp-json/indieauth/1.0/metadata，发行者 URL 可以是https://example.com/wp-json/indieauth/1.0

3.2用户个人资料网址

用户通过 [ URL ]进行标识。个人资料 URL必须具有https或http方案，必须包含路径组件（/是有效路径），不得包含单点或双点路径段，可以包含查询字符串组件，不得包含片段组件，不得包含用户名或密码组件，并且不得包含端口。此外，主机名必须是域名，不得是 ipv4 或 ipv6 地址。

以下是一些有效个人资料 URL 的示例：

  • https://example.com/
  • https://example.com/username
  • https://example.com/users?id=100

以下是一些无效个人资料网址的示例：

  • example.com- 缺少方案
  • mailto:user@example.com- 无效方案
  • https://example.com/foo/../bar- 包含双点路径段
  • https://example.com/#me- 包含片段
  • https://user:pass@example.com/- 包含用户名和密码
  • https://example.com:8443/- 包含一个端口
  • https://172.28.92.51/- 主机是 IP 地址

3.3客户端标识符

客户端通过 [ URL ]进行标识。客户端标识符 URL必须具有https或http方案，必须包含路径组件，不得包含单点或双点路径段，可以包含查询字符串组件，不得包含片段组件，不得包含用户名或密码组件，并且可以包含端口。此外，主机名必须是域名或环回接口，并且不得127.0.0.1是 IPv4 或 IPv6 地址（IPv4或 IPv6除外）[::1]。

3.4URL 规范化

由于 IndieAuth 使用 https/http URL，这些 URL 属于 [ URL ] 所称的“特殊 URL ”，因此没有路径部分的字符串不是有效的 [ URL ]。因此，如果遇到没有路径部分的 URL，则必须将其视为具有路径/。例如，如果用户提供https://example.comDiscovery，则客户端在使用和比较时必须将其转换为。https://example.com/

由于域名不区分大小写，因此 URL 的主机部分必须不区分大小写。在存储和使用 URL 时，实现应将主机转换为小写。

为了方便使用，客户端可以允许用户仅输入 URL 的主机部分，在这种情况下，客户端必须在开始 IndieAuth 流程之前将其转换为有效的 URL，方法是在前面添加http或https方案并附加路径/。例如，如果用户输入example.com，则客户端会http://example.com/在开始发现之前将其转换为。

4.发现

本规范使用 [ HTML ]定义的链接关系注册表，适用于 HTML 和 HTTP 链接关系。

4.1客户发现

当用户登录时，客户端需要发现一些信息。客户端需要发现用户的indieauth-metadata端点，该端点提供 IndieAuth 服务器授权端点和令牌端点的位置，以及客户端的其他相关信息。客户端必须首先对 [ Fetch ] 用户提供的 URL发出 GET 或 HEAD 请求以发现必要的值。客户端必须遵循 HTTP 重定向（最高可达自我设定的限制）。当使用授权流程获取用于另一个端点（例如 [ Micropub ] 端点）的访问令牌时，客户端还将发现该micropub端点。

客户端必须检查带有值的HTTPLink标头 [ RFC8288 ] 。如果文档的内容类型是 HTML，则客户端必须检查带有值的HTML 元素。如果存在多个这样的元素，则第一个 HTTP标头优先，然后是按文档顺序的第一个元素。relindieauth-metadata<link>relindieauth-metadataLink<link>

发现的 URL可能是相对 URL，在这种情况下，客户端必须根据 [ URL ] 相对于当前文档 URL 来解析它们。

客户端可以首先发出 HTTP HEAD 请求 [ RFC7231Link ] 来跟踪重定向，并在发出 GET 请求之前检查标头。

如果没有indieauth-metadata提供 URL，为了与 IndieAuth 的先前版本兼容，客户端应该按照与上面描述的相同优先顺序查找值为（和可选）的HTTPLink标头和 HTML<link>元素。relauthorization_endpointtoken_endpoint

4.1.1IndieAuth 服务器元数据

IndieAuth 元数据采用 OAuth 2.0 授权服务器元数据 [ RFC8414 ]，但显著的区别在于 URL 的发现是通过 IndieAuth 链接关系进行的，而不是通过.well-knownRFC8414 指定的发现方法。为了与其他 OAuth 2.0 实现兼容，建议.well-known使用RFC8414 中定义的路径，但这不是必需的。

元数据端点以 JSON 对象的形式返回有关服务器的信息，具有以下属性：

  • issuer- 服务器的发行者标识符。
  • authorization_endpoint- 授权端点
  • token_endpoint- 令牌端点
  • introspection_endpoint- 自省端点
  • introspection_endpoint_auth_methods_supported（可选） - 包含此内省端点支持的客户端身份验证方法列表的 JSON 数组。
  • revocation_endpoint（可选）- 撤销端点
  • revocation_endpoint_auth_methods_supported（可选）- 包含值的 JSON 数组"none"。如果提供了撤销端点，则还应提供此属性的值，因为根据 [ RFC8414 ]，["none"]省略此值默认为。client_secret_basic
  • scopes_supported（推荐）- 包含 IndieAuth 服务器支持的范围值的 JSON 数组。即使使用此参数，服务器也可以选择不公布某些支持的范围值。
  • response_types_supported（可选）- 包含支持的 response_type 值的 JSON 数组。这与 [RFC8414] 不同，因为此参数是可选的，如果省略，则默认为code
  • grant_types_supported（可选）- 包含支持的授权类型值的 JSON 数组。如果省略，默认值与 [RFC8414] 不同，并且authorization_code
  • service_documentation（可选）- 包含开发人员在使用服务器时可能需要了解的人性化信息的页面 URL。这可能是指向 IndieAuth 规范的链接，也可能是与您的实现相关的更个性化的内容。
  • code_challenge_methods_supported- 包含 PKCE 支持的方法的 JSON 数组。此参数与 [RFC8414] 不同，因为它不是可选的，因为 PKCE 是必需的。
  • authorization_response_iss_parameter_supported（可选）- 布尔参数，指示授权服务器是否提供该iss参数。如果省略，默认值为 false。由于该iss参数是REQUIRED，因此提供该参数是为了与实现该参数的 OAuth 2.0 服务器兼容。
  • userinfo_endpoint（可选）- 用户信息端点

5.授权

本节介绍如何使用 IndieAuth 的 OAuth 2.0 授权码流对用户进行身份验证并选择性地获取访问令牌。

  • 最终用户在客户端的登录表单中输入 URL，然后单击“登录”。客户端将规范化该 URL。
  • rel=indieauth-metadata客户端通过获取提供的 URL 并查找值来发现最终用户的 IndieAuth 服务器元数据端点
  • 客户端通过获取元数据 URL 并查找authorization_endpoint和token_endpoint值来发现服务器的授权端点和令牌端点
  • 客户端构建授权请求，包括其客户端标识符、请求的范围、本地状态和重定向 URI，并将浏览器重定向到授权端点
  • 授权端点从客户端标识符 URL 获取客户端信息，以便向用户显示应用程序名称和图标
  • 授权端点验证最终用户（例如通过登录）并确定最终用户是否同意或拒绝客户端的请求
  • 授权端点生成授权码并将浏览器重定向回客户端，并在 URL 中包含授权码
  • 客户端通过向令牌端点发出 POST 请求，将授权码换成访问令牌。令牌端点验证授权码，并使用最终用户的规范个人资料 URL 和访问令牌进行响应
  • 客户端确认返回的配置文件 URL 声明了相同的授权服务器并接受配置文件 URL

注意：如果客户端仅想了解用户是谁并且不需要访问令牌，则客户端会在授权端点处交换授权码以获取用户配置文件信息。

  “code_challenge_methods_supported”：[“S256”]
}

5.2授权请求

authorization_endpoint客户端从发现的URL 开始并向查询组件添加参数来构建授权请求 URL 。

所有 IndieAuth 客户端都必须使用 PKCE ([ RFC7636 ]) 来防止授权代码注入和 CSRF 攻击。下面介绍了 PKCE 机制的非规范描述，但实施者应参考 [ RFC7636 ] 了解详细信息。

客户端对每个授权请求使用一个唯一的密钥来防止授权码注入和 CSRF 攻击。客户端首先生成此密钥，然后可以将其与授权码一起使用，以证明使用授权码的应用程序与请求授权码的应用程序是同一个应用程序。

客户端首先为每个授权请求创建一个代码验证器，即使用最小长度为 43 个字符、最大长度为 128 个字符的字符生成一个随机字符串[A-Z] / [a-z] / [0-9] / - / . / _ / ~。此值存储在客户端上，稍后将在授权码交换步骤中使用。

然后，客户端通过计算代码验证器的 SHA256 哈希值并对结果进行 Base64-URL 编码，创建从代码验证器派生的代码挑战。

code_challenge = BASE64URL-ENCODE(SHA256(ASCII(code_verifier)))

为了向后兼容，如果授权服务器希望支持旧客户端，授权端点可以接受没有代码质询的授权请求。

  • response_type=code- 向授权服务器指示应返回授权码作为响应
  • client_id- 客户端 URL
  • redirect_uri- 重定向 URL，指示用户在批准请求后应重定向到的位置
  • state- 客户端设置的参数，当用户重定向回客户端时将包含该参数。这用于防止 CSRF 攻击。授权服务器必须将未修改的状态值返回给客户端。
  • code_challenge- 如前所述的代码挑战。
  • code_challenge_method- 用于计算代码挑战的哈希方法，例如“S256”
  • scope-（可选）客户端请求的范围的空格分隔列表，例如“profile”或“profile create”。如果客户端省略此值，授权服务器不得为此授权代码颁发访问令牌。只能返回用户的个人资料 URL，而无需请求任何范围。有关请求哪些范围以返回用户个人资料信息的详细信息，请参阅个人资料信息。
  • me- (可选) 用户输入的 URL
例子5 
https://example.org/auth?response_type=code&
                         client_id=https://app.example.com/&
                         redirect_uri=https://app.example.com/redirect&
                         状态=1234567890&
                         代码挑战=OfYAxt8zU2dAPDWQxTAUIteRzMsoj9QBdMIVEDOErUo&
                         代码挑战方法=S256&
                         范围=配置文件+创建+更新+删除&
                         我=https://user.example.net/

客户端应该向授权端点提供查询字符串参数，可以是用户输入的精确值，也可以是应用URL 规范化me后的值。

授权端点应该获取client_idURL 来检索应用程序信息和客户端的注册重定向 URL，有关更多信息，请参阅客户端信息发现。

redirect_uri如果请求中的URL 方案、主机或端口与 的不匹配client_id，则授权端点应该验证请求是否与客户端发布的重定向 URLredirect_uri之一匹配，如果不匹配，则应该阻止该请求继续进行。

如何对用户进行身份验证取决于授权端点。此步骤超出了 OAuth 2.0 的范围，并且高度依赖于特定的实现。一些授权服务器使用典型的用户名/密码身份验证，而其他授权服务器则使用其他形式的身份验证（例如 [ RelMeAuth ]），或委托给其他身份提供者。

授权端点可以使用提供的me查询组件来提示哪个用户正在尝试登录，并指示客户端在生成的个人资料 URL 响应或访问令牌响应中期望哪个个人资料 URL。这对于用户拥有多个受支持的个人资料 URL 的授权端点特别有用，因此授权端点可以做出明智的决定，确定用户想要识别为哪个个人资料 URL。请注意，从授权端点的角度来看，客户端提供的这个值是未经验证的外部数据，在此阶段不得被视为有效数据。如果登录用户me与客户端提供的参数不匹配，授权端点可以完全忽略该me参数或显示错误，由授权端点自行决定。

用户通过身份验证后，授权端点会向用户显示授权请求。提示必须指明用户正在登录哪个应用程序，并且应提供有关请求的尽可能详细的信息，例如有关请求范围的信息。

5.2.1授权响应

如果用户批准请求，授权端点将生成授权码并重定向回客户端。

重定向的构建从redirect_uri请求中的开始，并将以下参数添加到重定向 URL 的查询组件：

  • code- 授权端点生成的授权码。该代码必须在发出后不久过期，以降低泄露风险，并且必须只能使用一次。建议最长使用时间为 10 分钟。有关授权码的其他要求，请参阅OAuth 2.0 第 4.1.2 节。
  • state- 状态参数必须设置为客户端在请求中设置的准确值。
  • iss- 用于客户端验证的发行者标识符。
例子6 
HTTP/1.1 302 已找到
位置：https://app.example.com/redirect?code=xxxxxxxx&
                                           状态=1234567890&
                                           iss=https%3A%2F%2Findieauth.example.com

重定向回客户端后，客户端必须验证：

  • 请求中的参数state有效并且与最初创建的状态参数匹配，以防止 CSRF 攻击。状态值还可以存储会话信息，以便开发无法自行存储数据的客户端。
  • iss请求中的参数有效，并与发现期间服务器元数据端点提供的颁发者参数匹配，如 OAuth 2.0 授权服务器颁发者标识 [RFC9207] 中所述。客户端必须使用简单的字符串比较来比较参数。如果值与预期的颁发者标识符不匹配，客户端必须拒绝授权响应，并且不得继续授权。对于错误响应，客户端不得假设错误源自预期的授权服务器。

请参阅 OAuth 2.0 [ RFC6749 ]第 4.1.2.1 节，了解如何向用户和客户端指示错误和其他失败。

5.3兑换授权码

一旦客户端获得授权码，它就可以将其兑换为访问令牌或用户的最终个人资料 URL。

5.3.1要求

如果客户端需要访问令牌才能向资源服务器（如 [ Micropub ] 端点）发出请求，则可以在令牌端点处使用授权码交换访问令牌和用户的个人资料 URL 。

如果客户端只需要知道登录的用户，而不需要使用访问令牌向资源服务器发出请求，则客户端在授权端点用授权码交换用户的个人资料 URL 。

客户端验证状态参数后，客户端向令牌端点或授权端点发出 POST 请求，以将授权代码交换为最终用户个人资料 URL 和/或访问令牌。POST 请求包含以下参数：

  • grant_type=authorization_code
  • code- 从重定向中的授权端点收到的授权码。
  • client_id- 客户端的 URL，必须与身份验证请求中使用的 client_id 匹配。
  • redirect_uri- 客户端的重定向 URL，必须与初始身份验证请求相匹配。
  • code_verifier- 开始授权请求之前生成的原始明文随机字符串。

发布 https://indieauth.example.com/auth
内容类型：application/x-www-form-urlencoded
接受：application/json

grant_type=授权码
&代码=xxxxxxxx
&client_id=https://app.example.com/
&redirect_uri=https://app.example.com/redirect
&代码验证器=a6128783714cfda1d388e2e98b6ae8221ac31aca31959e59512c59f5
令牌端点请求示例
例子8 
发布 https://indieauth.example.com/token
内容类型：application/x-www-form-urlencoded
接受：application/json

&代码=xxxxxxxx
&client_id=https://app.example.com/
&redirect_uri=https://app.example.com/redirect
&代码验证器=a6128783714cfda1d388e2e98b6ae8221ac31aca31959e59512c59f5

请注意，为了向后兼容，授权端点可以允许没有 的请求code_verifier。如果授权码是在没有code_challenge存在的情况下发出的，则授权码交换不得包括code_verifier，同样，如果授权码是在code_challenge存在的情况下发出的，则授权码交换必须包括code_verifier。

5.3.2个人资料 URL 响应

如果客户端只需要知道登录的用户，则客户端将在授权端点交换授权码，并且仅返回规范的用户配置文件 URL 和可能的配置文件信息。

授权端点会验证授权代码是否有效、是否尚未使用，以及是否为匹配的client_id和颁发redirect_uri，并检查提供的code_verifier哈希值是否与原始授权请求中的 相同code_challenge。如果请求有效，则端点会使用 JSON [ RFC7159 ] 对象进行响应，其中包含属性me，以及登录用户的规范用户个人资料 URL，以及可选的属性，profile其中包含个人资料信息中定义的用户个人资料信息。

例子9 
HTTP/1.1 200 正常
内容类型：application/json

{
  “我”：“https://user.example.net/”
}

生成的配置文件 URL可能与提供给客户端进行发现的 URL 不同。这为授权服务器提供了规范化用户 URL 的机会，例如更正为http，https或在必要时添加路径。请参阅不同的用户配置文件 URL，了解客户端开发人员应注意的安全注意事项。

请参阅 OAuth 2.0 [ RFC6749 ]第 5.2 节，了解在发生错误或其他故障时如何响应。

5.3.3访问令牌响应

当客户端收到使用一个或多个范围请求的授权码并将导致返回访问令牌时，客户端将在令牌端点交换授权码。

client_id令牌端点需要验证授权码是否有效，是否为匹配的和颁发redirect_uri，是否包含至少一个scope，并检查提供的code_verifier哈希值是否与code_challenge原始授权请求中 给出的值相同。如果授权码未与 一起颁发scope，则令牌端点不得颁发访问令牌，因为根据 OAuth 2.0 [ RFC6749 ]第 3.3 节，空范围无效。

令牌端点如何验证授权码的细节超出了本文档的范围，因为通常授权端点和令牌端点是同一系统的一部分，并且可以共享存储或其他私有通信机制。

如果请求有效，则令牌端点可以生成访问令牌并返回适当的响应。令牌响应是一个 JSON [ RFC7159 ] 对象，包含：

  • access_token（必需） - OAuth 2.0 Bearer Token [ RFC6750 ]。
  • me（必需） - 此访问令牌对应的用户的规范用户个人资料 URL。
  • profile（可选）- 用户的个人资料信息，如个人资料信息中定义。
  • expires_in（推荐） - 访问令牌的有效期（以秒为单位）。
  • refresh_token（可选） - 刷新令牌，可用于获取刷新令牌中定义的新访问令牌。

例如：

例子10 
HTTP/1.1 200 正常
内容类型：application/json

{
  “访问令牌”：“XXXXXX”，
  "token_type": "持有者",
  “范围”：“创建更新删除”，
  “我”：“https://user.example.net/”
}

生成的配置文件 URL可能与提供给客户端进行发现的 URL 不同。这为授权服务器提供了规范化用户 URL 的机会，例如更正为http，https或在必要时添加路径。请参阅不同的用户配置文件 URL，了解客户端开发人员应注意的安全注意事项。

请参阅 OAuth 2.0 [ RFC6749 ]第 5.2 节，了解在发生错误或其他故障时如何响应。

5.3.4个人资料信息

请求个人资料信息

如果客户端除了确认用户的个人资料 URL 之外还想请求用户的个人资料信息，则客户端可以在初始授权请求中包含一个或多个范围。scope本规范定义了以下值来请求有关用户的个人资料信息：

  • profile（必需）——此范围请求访问用户的默认个人资料信息，其中包括以下属性：name，，photo。url
  • email- 此范围请求访问以下属性中的用户电子邮件地址：email。

请注意，由于profile请求配置文件信息时需要范围，因此email不能单独请求范围，而必须profile根据需要与范围一起请求。

当授权码以上述任何范围发布时，交换授权码时的响应可能包含一个新属性，即 ，与授权端点或令牌端点响应中的属性profile一起。该属性定义为 JSON [ RFC7159 ] 对象，具有上述每个范围定义的属性。meprofile

例如，对具有范围的请求的完整响应profile email create（包括访问令牌和配置文件信息）可能如下所示：

例子11 
HTTP/1.1 200 正常
内容类型：application/json

{
  “访问令牌”：“XXXXXX”，
  "token_type": "持有者",
  "scope": "创建个人资料电子邮件",
  "me": "https://user.example.net/",
  “轮廓”： {
    “名称”：“示例用户”，
    “url”：“https://user.example.net/”，
    "photo": "https://user.example.net/photo.jpg",
    “电子邮件”：“user@example.net”
  }
}

与 OAuth 2.0 的情况一样，无法保证客户端请求的范围将由授权服务器或用户授予。即使在请求配置文件范围时，客户端也不应该依赖于配置文件信息的存在。因此，实现从授权服务器返回配置文件信息的支持完全是可选的。

对象中返回的信息profile仅供参考，无法保证这些信息是“真实的”或“经过验证的”。提供的信息只是用户选择与客户端共享的信息，甚至可能因请求此数据的客户端而异。

  • name- 用户希望向客户提供的姓名。客户不应将此视为用户的全名。客户应将其用作显示名称。
  • url- 用户网站的 URL。url不保证与 URL 匹配me，甚至可能有不同的主机。例如，多作者网站可能使用网站的 URL 作为 URL me，但在个人资料数据中返回每个特定作者自己的个人网站。
  • photo- 用户希望客户用作个人资料图片的照片或图像。
  • email-（如果请求电子邮件范围）用户希望向客户提供的电子邮件地址。

客户端不得将对象中的信息profile视为规范或权威的信息，也不得以该信息为基础做出任何身份验证或识别决定。

例如，尝试使用配置email文件对象中返回的作为用户标识符将导致安全漏洞，因为任何用户都可以创建一个授权端点，该端点在配置文件响应中返回任何电子邮件地址。使用此处返回的电子邮件地址的客户端应将其视为已在客户端应用程序中手动输入的地址，并在使用它之前完成自己的验证过程。

5.4授权服务器确认

客户端最初会提示用户输入 URL，以便发现执行身份验证或授权所需的端点。但是，用户最初输入的 URL 与授权服务器返回的最终配置文件 URL 之间可能存在差异。这些差异可能是不同的方案（http 与 https），甚至是具有不同主机的 URL。

在从授权服务器接收到me响应中的 URL（在配置文件 URL 响应或访问令牌响应中）后，客户端必须验证授权服务器是否有权对返回的配置文件 URL 提出声明，方法是确认返回的配置文件 URL 声明了相同的授权服务器。

客户端必须对返回的URL执行端点发现me，并验证 URL 声明的授权端点与初始发现步骤中发现的相同，除非返回的URL 与最初输入的 URL 或在初始端点发现me期间遇到的任何 URL 完全匹配，无论是来自可能的重定向链还是作为最终值。

请注意，在初始端点发现中检查返回的配置文件 URL 是否存在的步骤是一个可选的优化步骤，这可以节省客户端可能需要发出另一个 HTTP 请求的时间。为简单起见，可以跳过此步骤，因为从返回的配置文件 URL 发现授权服务器足以确认返回的配置文件 URL 声明了相同的授权服务器。

此验证步骤可确保授权端点无法对任意配置文件 URL 发出有效响应，并且共享域上的用户不能代表该域的其他用户伪造授权。

  5. 客户端向授权端点发出 GET 请求https://example.com/并找到该端点，https://login.example.com
    • 注意：服务器也可以直接在请求响应中公布授权端点，http://user@example.com/而不需要单独的重定向
  6. 客户端使用 执行 IndieAuth 流程https://login.example.com，将用户user@example.com在请求中输入的内容作为提示提供给服务器。这会导致配置文件 URL 响应中的值为 ，me作为https://example.com/username规范配置文件 URL。
  7. 这是客户端第一次看到此 URL，因此必须验证此子域与授权服务器之间的关系。它获取https://example.com/username并找到相同的授权端点https://login.example.com
  8. 客户端接受me的值https://example.com/username

5.5刷新令牌

刷新令牌由授权服务器颁发给客户端，并且可以随时使用以获取新的访问令牌，通常是在当前访问令牌无效或过期时，或者获取具有相同或更窄范围的新令牌（访问令牌的生命周期可能比资源所有者授权的更短，权限也更少）。

建议使用短期访问令牌并提供刷新令牌，但是，是否颁发刷新令牌由授权服务器自行决定，可能基于客户端的属性、请求的属性、授权服务器内的策略、授权请求的用户的选择或任何其他标准颁发。如果授权服务器颁发刷新令牌，则在颁发访问令牌时将其包含在返回中。如果授权服务器决定不颁发刷新令牌，或者刷新令牌已过期，则客户端可以通过重新启动授权流程来获取新的访问令牌。

如果发生安全事件（例如密码更改或授权服务器注销），授权服务器可以自动撤销刷新令牌，或者在兑换刷新令牌时，可以提供新的刷新令牌。如果客户端在一段时间内处于非活动状态，即在一段时间内未使用刷新令牌获取新的访问令牌，则刷新令牌应该过期。到期时间由授权服务器自行决定。

5.5.1刷新访问令牌

要刷新访问令牌，客户端需要向令牌端点发出 POST 请求，以将刷新令牌换成新的访问令牌。POST 请求包含以下参数：

  • grant_type=refresh_token
  • refresh_token- 先前向客户端提供的刷新令牌。
  • client_id- 颁发刷新令牌时使用的客户端 ID。
  • scope（可选）- 客户端可以请求具有与原始访问令牌相同或更少范围的令牌。如果省略，则视为等于授予的原始范围。

例如：

例子12 
发布 https://example.org/token
内容类型：application/x-www-form-urlencoded
接受：application/json

grant_type=refresh_token
&refresh_token=xxxxxxxx&client_id=https://app.example.com

如果有效且获得授权，授权服务器将发出访问令牌，如访问令牌响应中所述。授权服务器可以发出新的刷新令牌，在这种情况下，客户端必须丢弃旧刷新令牌并将其替换为新刷新令牌。授权服务器可以在向客户端发出新刷新令牌后撤销旧刷新令牌。如果发出了新的刷新令牌，则刷新令牌范围必须与客户端在请求中包含的刷新令牌范围相同。

如果客户端一段时间内未使用刷新令牌获取新的访问令牌，则刷新令牌应该过期。过期时间由授权服务器决定。

6.访问令牌验证

在 OAuth 2.0 中，访问令牌对客户端是不透明的，因此客户端不需要知道有关令牌本身的内容或结构的任何信息。此外，客户端向其发出请求的端点（例如 [ Micropub ] 端点）甚至可能不了解如何解释由独立令牌端点颁发的令牌。如果令牌端点未与客户端正在交互的资源服务器紧密集成，则资源服务器需要一种方法来验证从客户端收到的访问令牌。如果令牌端点和 Micropub 端点紧密耦合，那么它们当然可以使用内部机制来验证访问令牌。

旨在与其他端点互操作的令牌端点必须使用下面描述的机制来允许资源服务器（例如 Micropub 端点）验证访问令牌。

6.1访问令牌验证请求

如果资源服务器需要验证访问令牌是否有效，则可以使用令牌自检来进行验证。IndieAuth 扩展了 OAuth 2.0 令牌自检 [ RFC7662 ]，并规定自检响应必须包含一个附加参数me。

请注意，对端点的请求将不包含任何用户识别信息，因此资源服务器（例如 Micropub 端点）将需要通过带外方法知道正在使用哪个令牌端点。

资源服务器应向令牌端点发出 POST 请求，并在token参数中包含 Bearer 令牌，这将生成令牌验证响应。端点还必须要求某种形式的授权才能访问此端点，并可在元数据响应的参数中标识这一点introspection_endpoint_auth_methods_supported。如果授权不足以满足请求，授权服务器必须以 HTTP 401 代码进行响应。

例子十三 
发布 https://indieauth.example.com/introspect
内容类型：application/x-www-form-urlencoded
接受：application/json
授权：持有人 xxxxxxxx

令牌=xxxxxxxx

6.2访问令牌验证响应

令牌端点验证访问令牌（如何进行验证取决于实现），并返回有关令牌的信息：

  • active-（必需）布尔指示器，指示所呈现的令牌当前是否处于活动状态
  • me- (必填) 此 token 对应的用户资料 URL
  • client_id- 与此令牌关联的客户端 ID
  • scope- 与此令牌关联的范围的空格分隔列表
  • exp- (可选) 整数时间戳，以 UTC 1970 年 1 月 1 日以来的秒数计算，表示此令牌何时过期
  • iat- (可选) 整数时间戳，以 UTC 1970 年 1 月 1 日以来的秒数来衡量，表示此令牌最初发行的时间
例子14 
HTTP/1.1 200 正常
内容类型：application/json

{
  “活跃”：“真”，
  "me": "https://user.example.net/",
  “客户端 ID”：https://app.example.com/”，
  “范围”：“创建更新删除”
  “exp”：“1632443647”，
  “iat”：“1632443147”
}

具体实现可以将附加参数作为顶级 JSON 属性。客户端应忽略它们无法识别的参数。

如果令牌无效，端点仍必须返回 200 响应，其中唯一参数为 active（其值设置为“false”）。响应不应包含有关非活动令牌的任何其他信息，包括令牌非活动的原因。

例子15 
HTTP/1.1 200 正常
内容类型：application/json

{
  “active”：“false”，
}

7.1令牌撤销请求

以下是撤销请求的示例。

例子16 
POST https://indieauth.example.com/revocation HTTP/1.1
内容类型：application/x-www-form-urlencoded

如 [ RFC7009 ]中所述，当令牌被成功撤销或提交的令牌无效时，撤销端点都会以 HTTP 200 进行响应。

8.访问受保护的资源

客户端通过向资源服务器出示访问令牌来访问受保护的资源。资源服务器必须验证访问令牌，并确保其未过期且其范围涵盖所请求的资源。

8.1错误响应

当请求失败时，资源服务器使用适当的 HTTP 状态代码进行响应，并在响应中包含以下错误代码之一：

  • "invalid_request"- 请求无效。资源服务器应以 HTTP 400 响应
  • "invalid_token"- 提供的访问令牌已过期、已撤销或无效。资源服务器应以 HTTP 401 响应
  • "insufficient_scope"- 请求需要的权限高于提供的权限。资源服务器应该以 HTTP 403 响应

如果请求缺少任何身份验证信息，资源服务器不应包含错误代码或其他信息。

9.用户信息

客户端可能希望刷新或接收授权响应之外的有关经过身份验证的最终用户的其他信息。为了获取用户的个人资料信息，客户端向 userinfo 端点发出 GET 请求，并提供使用profile和/或email范围颁发的访问令牌。

例子17 
获取/用户信息 HTTP/1.1
主机：indieauth.example.com
授权：持有人 xxxxxxxxxxx

当使用的值导致发出访问令牌并且需要相同或范围时，授权服务器将返回与配置文件信息中标识的配置文件属性相同的JSON [ RFC7159 ] 对象。配置文件信息中确定的有关信息非权威性的注意事项也适用于此。response_typeprofileemail

如果请求缺少提供的访问令牌，或者令牌不包含适当的范围，则端点应该以错误响应进行响应，如访问受保护的资源中所述。

与授权响应中返回的个人资料信息一样，userinfo 端点的实现完全是可选的。如果实现，则发现将通过userinfo_endpoint授权服务器元数据文档中的属性进行。

例子18 
HTTP/1.1 200 正常
内容类型：application/json

{
    “名称”：“示例用户”，
    “url”：“https://user.example.net/”，
    "photo": "https://user.example.net/photo.jpg",
    “电子邮件”：“user@example.net”
}

10.安全注意事项

除了 OAuth 2.0 核心 [ RFC6749 ] 和 OAuth 2.0 威胁模型和安全注意事项 [ RFC6819 ] 中的安全注意事项之外，还需考虑其他事项。

10.1防止网络钓鱼和重定向攻击

授权服务器应获取client_id身份验证或授权请求中提供的信息，以便向用户提供有关请求的其他信息，例如应用程序名称和徽标。如果服务器未获取客户端信息，则应采取额外措施，确保向用户提供有关请求的尽可能多的信息。

授权服务器应在授权界面上显示完整信息client_id，并显示获取的应用程序信息（如果有）。显示信息client_id可帮助用户知道他们正在授权预期的应用程序。

由于 IndieAuth 客户端很可能是公共客户端（如果jwks_uri客户端元数据中未公布），因此唯一可用于防范 [ RFC6819 ] 中描述的某些攻击的措施是对客户端的进行强验证redirect_uri。如果redirect_uri方案、主机或端口与不同client_id，则授权服务器必须按照重定向 URL中所述验证重定向 URL ，或向用户显示重定向 URL，以便他们手动检查。
