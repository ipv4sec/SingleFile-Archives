继承被新语言抛弃的原因是这玩意儿太强了, 经常会搞出出人意料的奇异现象, 人类把握不住.

比起继承带来的问题, 其好处直接被淹没了, 还不如用弱点的组合.

代码多写几行, 问题少亿点点.

含继承和子类型的语言里你要知道如下概念, 什么最小知识原则, 什么开闭原则, 什么里氏替换原则, 什么逆变协变双变抗变, 什么构造函数顺序地狱, 什么多的记不住的访问权限关键词, 什么多继承菱形继承虚基类, 什么 C3 线性化过程....

不过这些都是实践中的小小问题罢了, 花点心思总能解决.

致命问题是来自于整个子类型理论的, 92 年 Pierce 证明了有界量化不可判定[1], 基于 F<:F_{<:} 的语言傻了, 后面 Levy 和 Tiuryn 又独立证明了二阶类型不可判定[2], 基于 Π\Pi 类型的语言也傻了, 学术型语言彻底放弃引入子类型, 而学术型语言正是工业型语言的风向标.

你路尽头都被这诡异的子类型污染了你还修个啥啊, 越修越诡异.

继承的前提是 is-a，男人是人，男人可以从人的class中继承，机器人不是人，哪怕有手有脚，也不能从人继承。

代码里如何知道是不是 is-a？要看父类及其父类的所有公共接口，是不是都符合自己的需求，甚至要考察一下父类是怎么被调用的，以符合里氏替换的情况。

而从第三方代码继承时，基本很难做到了解父类的所有接口和应用场景，在还不知道父类是什么的时候，就声明了 is-a 的关系。这就埋下了隐患。

继承是一个强有力的承诺，不了解就承诺，其实是在滥用。

但继承不是一无是处，我的体会是，当父类代码在自己的控制范围内，子类是 is-a 关系时，就大胆的去用继承，往往比组合自然优雅。

在很多的库中，继承也非常常见。继承不是洪水猛兽，不要滥用，也不要不用，实事求是就好。

等你在这个结构上写了几万行代码，策划对你说：“塔要加个护盾技能，可以短暂无敌，小兵要加个合体技能，3个小兵合体成一个大兵。”

继承关系一下就被破坏了，相当于动了地基，大量痛苦的代码重构必不可少。

除了破坏父类，策划还可能提出这些需求：实现只会走路的单位，只会闪现的单位，会走路、闪现的单位，会闪现、攻击的单位。

这些功能你明明都写过代码，此时却无法复用，只能不甘心的重构代码。

归根结底，游戏行业是个创意行业，策划的各种设定从来就是不稳定的，各种天马行空，靠继承很难搞定。

很合理吧，充分利用重复代码的同时，又代码间各自独立。

先给出几个忠告：

  1. 不能拿学校的教学用例来当作现实。因为学校的教学用例，跟现实的匹配程度，基本上和中学物理题里的“忽略一切阻力……”差不多。
  2. 不要忽视重构的成本、难度和风险。你在学校里写的代码，其复杂度完备程度，甚至比不上真实工作中的一个demo水平，所以重构一下也无所谓，说不定还能学到点东西——最起码也能混个手熟。而在现实中，就算真的有宣称可以“安全重构的IDE”（我个人表示极度怀疑到底有没有这么嚣张的IDE——哪怕仅仅是宣传）。然而用它重构完后，你敢不测试就发布吗？那既然要测试，我就光说测试：有多少用例能直接用原来的？有多少需要重新适配？有多少甚至基本不能用要完全重写的？有多少新出现的场景和流程是现有用例完全没覆盖，需要全新设计和开发的？最后还有多少是场景已经不存在，整个用例都要拿掉的？一个普普通通的中小型项目，测试完备点的，用例数动辄几千几万，按照这标准全部review和评审一遍，你觉得要多久？就这我还很友善的假定了你的用例不涉及打桩和你的重构不涉及打桩模式的改动，要不然，工作量起码再往上加一大堆。
  3. 学校里写的代码，其工作量和交付时间的比值，要比真实工作低了好几个数量级。所以，不少你觉得无所谓的事情，到了真实工作中那都是天大的麻烦（例如说你刚才提到的重构）。UML/面向对象分析这类东西，不是说不能用，但在高强度需求压力，高频度小步快跑，高密度滚动迭代的场景下，几乎没有团队会正儿八经的搞这套流程。


所以，OO这玩意，最大的问题是它应对变化的灵活性太差。通常只有在业务高度稳定的领域（通常还要再加一个条件：不差钱也不差人），才值得花大精力去做什么“UML/面向对象分析”，值得一个个对象去研究：它到底是is-a还是has-a甚至仅仅是like-a。或者它is-a还是is-b？尤其是在整个继承树高度很大的时候，稍微在靠近基干的地方动一下，基本上可以认为后续绝大多数的对象都要全部重新这么审视一遍——也就是说，在OO模式下，对象的变动是会扩散的。然而，在还没有稳定的业务中，很多东西根本说不清楚，回答不了上述问题。或者这些回答三天两头就变。

如此一来，倒还不如老老实实先做一堆正交或者不那么正交的稳定的基础构件，然后一个劲的搭积木。今天你要a我就给你搭a，要b我就给你搭b，明天你说不要a了，我就把a摘下来（或者简单的屏蔽一下）。到了测试也差不多，基础构件认真测，保证质量。搭积木的只要把组合逻辑跑一下，基本上也就出不了什么大事了。毕竟积木之间相互不依赖，所以变动不会扩散。当然，你要是说基础构件也不稳定，也要经常变？那在动手改代码之前，真诚建议你先去更新一下你的简历，以防万一。。。

一无是处是不可能的，要是真一无是处，这些用了class的语言的流行难道是因为天生是OO教徒？

继承这套思想显然在上个世纪末就非常流行了，尤其是C++和Java，当然，Java没抄明白，非要学Smalltalk，这是后话。C++用模板和继承来实现元编程，也是后话。

根本原因在于继承的目的是为了解决工程问题，但是后来人们把继承的用途扩大化了。

回想C和C++发明的年代，图形界面基本是不存在的，显示器的横向像素数量都非常有限，现在很多人所谓继承适合GUI编程，其实没太大道理。

可以参考C++他爹Bjane Strousrup的《C++语言的设计和演化》，这本书对他设计的所有特性都有解释：

带类的C的成员函数和访问控制都是学的simula，引入这些功能显然使得程序员更安全的操作结构体。很多习惯C的人常常忘记初始化结构体成员，或者对结构体成员进行误用，造成了非常多的错误。

带类的C不像后来的C++，带类的C没有虚函数，Bjarne怀疑自己是否有人能教会人们如何使用它。即使没有虚函数，派生类也很有用，可以用于从来的类构造出新的数据结构，用于将操作和结果类型关联起来，特别是能够定义各种表类和作业类[Stroustrup 1980，1982b]。

[Stroustrup 1982b]给出了一段很难看的代码，用来完成多态，这段代码对类型进行枚举作为表，以便对指针进行不同操作。带类的C和C++里所做的许多努力就是保证程序员不再去写这种代码。实际上13年后的MFC，乃至2022年的今天，C语言仍需使用这种代码。

继承的另外一个作用就是希望通过基类的组合，提供通用的容器类。书里展示了一种通过派生类成员函数嫁接基类实现的方法。



实际上到这，带类的C部分就没有了，C++之父借用成员函数和继承的目的就到此为止。带类的C引入成员函数的目的是为了避免手写函数表，引入继承的原因是为了复用基类代码。实际上上个世纪计算机的内存和硬盘大小都有限，是根本不存在，也不可能存在复杂性。

4年后，Bjane Strousrup又从simula中继承了虚函数，[Stroustrup 1986, 1986b]，给出的理由也很简单：

虚函数使得程序员不用再手动编写函数表。
抽象类不需要类的实现，换句话说只需要有抽象类的定义就能编译，这使得我们能将抽象类和实现进行划分。

你会发现，当初的设计并没有考虑过于复杂的情况，换句话说，现如今对于继承和虚函数的“滥用”反而是被发现的，是需求创造了这些复杂设计。

有这个疑问，是编程能力升级的必经之路。

如果不用继承，而是用接口加组合，代码就能自动变好吗？不会的，因为在完全暴露被包含类的接口的情况下，它俩是可以互换的。继承只是把组合冗长的语法简化了。

刚才说到完全暴露，其实这才是关键所在。继承就是对外界声明，你实现了基类的全部功能。现实中继承的大部分误用，都是因为子类只想实现部分功能，或者只是想复用某些逻辑。大白话就是，只是想借点力，并不是想承担基类的责任。

举个反例，你有一个类叫出货清单，它的职责就是存储、列出这份订单到底从仓库里出了多少货；当然根据具体需求，或许还得支持排序、计价等功能。这时你一看，让它继承了 list 类，这样好不好呢？事实上你不想别人把你的出货清单当成数组使用，你不想承担数组的责任，你只是在用数组实现功能，恰好有部分功能数组也能用罢了。

举个正例，你要写个带汇总的出库清单类，这东西完全可以当出库清单用，而且它还支持快速的统计，所以你继承了上述类，这有没有问题呢？我认为是没有硬伤的，这只是个人口味问题。现在我们耳濡目染，下意识地规避继承，如果让我来写，我其实大概也会写成包含的形式。

有人反驳：如果改动了基类的公开行为（public api），子类也会被改变，像病毒一样传染！万一引入新 bug 呢？这太可怕了！

那么，难道改成组合就不影响了么？难道那些普通地使用这个类的代码就不影响了么？这些都是依赖，有依赖就有问题。归根结底还是要遵守开放闭合原则——一份逻辑的职责一但确定，就不能修改原有职责，只能新增职责。这样，旧代码才不会不小心崩掉。

继承最亮眼的作用之一，就是在除了继承和接口没什么好用的形式化工具的语言里发挥强类型的好处。类型是一种命题，从编译器类型检查的层面出发自动验证业务逻辑，是测试的上位替代。比如清单中的姓名、时间、货名、计量单位、货源、地址、货管姓名等都可以用 String 表示，但这样就存在不小心把姓名和货源参数搞反的风险。弱类型语言如 js，只能通过测试来自动检查。但在 java 这样的语言里，你就可以用不同类型表示。由于 String 是不可变的，所以姓名类完全可以承担 String 的职责——存储和展示一段文字。

软件是为了解决人的需求而存在的，既包含终端用户的需求，也包含程序员自己的需求。如果这个东西能让自己少犯错、能方便理解，那就是好的；如果用了犯错更多了，更不好理解了，就是坏的。由于之前业界受继承的伤比好处多得多，所以大风气很排斥它。但人对事物的认识总是通过否定之否定回到客观的准绳上来，最终，还是要从具体需要出发来决定是否继承。

最近对比 Trait 系统：

  1. Trait 是开放的，你可以为第三方类实现 Trait，但你不能为第三方类实现接口。
  2. Trait 是可算术的，你可以设定 A + B + !C，但一些语言里这种算术不太方便，都得上泛型。

我认为 Trait 是 Subtyping 的上位替代，继承的好处几乎都可以在 Trait system 中得到体现。而且，继承的标准比较模糊，不利于实际应用，实践起来发现大部分时候还不如全用 Interface + Composition 来得方便。
