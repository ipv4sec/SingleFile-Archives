OAuth 2.0 重定向 URI 验证

为什么我们要求客户端在 OAuth 2.0 中使用授权码交换访问令牌时包含重定向 URI（第 4.1.3 节）？

请考虑以下情形：

  1. 恶意用户使用授权码授予类型流程在合法客户端上启动 OAuth 流程。
  2. 客户端将恶意用户重定向到授权服务器，包括客户端上恶意用户账户的状态信息。
  3. 恶意用户获取授权端点 URI 并将重定向更改为其恶意站点。
  4. 恶意用户诱骗受害者用户点击链接并授权访问（使用网络钓鱼或其他社会工程方法）。
  5. 受害用户认为这是一个有效的授权请求（看起来合乎道德），并授权访问。访问权限被授予正确的合法客户端。到目前为止一切正常。
  6. 授权服务器认为它正在将受害用户发送回客户端，但是由于重定向 URI 已更改，受害用户被发送到恶意网站。
  7. 恶意用户获取授权码并通过构建原始正确的重定向 URI 将其返回给客户端。
  8. 客户端用代码交换访问令牌，并将其附加到恶意用户的帐户。
  9. 恶意用户现在可以通过他的客户帐户访问受害者用户的数据。

其工作方式是，攻击者无法直接访问受保护的资源，但它会诱骗客户端将受害者的访问令牌附加到攻击者的帐户。

预先注册重定向 URI 可以大有帮助，但取决于匹配规则。由于许多大型提供商都有开放的重定向器，攻击者可以利用它们构建通过授权服务器验证的重定向 URI。

强烈建议要求客户端注册其完整的重定向 URI，而不允许任何变化或部分匹配。
